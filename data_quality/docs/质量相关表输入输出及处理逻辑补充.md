# 质量相关表输入、输出及处理逻辑补充说明

## 一、适用说明

本补充说明用于填充《2606v2-law数据资源盘点报告模板-质量.docx》中“流程9：数据质量分析”及与质量评价相关的清单表。这里的“表”既包括盘点报告中的业务清单表，也包括质量评价模块落地时需要维护的逻辑数据表。

需要特别说明：当前质量评价模块中，**规则库采用 JSON Spec 存储**，即 `data_quality/specs/quality_rule_library.spec.json`；而“数据源”指待评价数据的来源，例如 JSON 导入、CSV 导入、MySQL、Oracle、Hive 或后续 Java/OM 项目接入的数据表。二者不是同一概念。

## 四、可直接补入模板“流程9：数据质量分析”的内容

### 【输入】

1. 数据资源基本信息：数据资产名称、业务域、数据来源、数据形态、数据规模、数据责任人。
2. 数据字典：数据表清单、字段清单、字段类型、字段长度、字段描述、示例值、是否主键、是否必填、枚举值。
3. 质量评价规则库：六维质量规则、规则编号、规则名称、适用对象、检查要求、阈值、强弱校验级别、整改建议。
4. 待评价数据：样例数据或真实导入数据，支持 JSON、CSV、数据库表或后续 Java/OM 项目接入的数据。
5. 规则推荐与参数配置：推荐规则清单、人工确认结果、通过率阈值、责任角色、任务周期、扫描方式。
6. 质量执行结果：规则通过率、异常记录数、失败规则、异常样例、任务执行批次。
7. 问题溯源数据：真实血缘关系、源表源字段、目标表目标字段、ETL 任务、下游报表/API。

### 【处理逻辑】

1. 建立质量评价范围：根据数据资源基本信息和数据字典，确定本次评价覆盖的数据源、库、表、字段、业务域和批次。
2. 识别字段质量特征：结合字段名称、字段类型、字段长度、样例值、枚举值和业务说明，识别手机号、身份证号、邮箱、金额、日期、状态、编码等典型字段。
3. 匹配六维质量规则：从规则库中按规范性、完整性、准确性、一致性、时效性、可访问性匹配适用规则。
4. 生成并确认规则推荐：系统根据元数据和样例数据推荐规则，人工确认后可调整通过率、是否必填、数字位数、金额范围、时间阈值等业务参数。
5. 执行质量评价任务：按照已确认的规则参数执行质量校验，统计总记录数、失败记录数、规则通过率和异常样例。
6. 形成质量问题清单：将失败规则转化为质量问题，按问题维度、问题类型、严重程度、责任角色和整改建议进行归类。
7. 开展问题溯源分析：对接真实血缘数据后，定位问题来源系统、来源表字段和加工环节，同时分析受影响的下游报表、接口或业务应用。
8. 计算六维质量得分：根据规则执行结果、维度权重和规则级别计算规范性、完整性、准确性、一致性、时效性、可访问性得分及总分。
9. 输出质量分析结论：汇总质量评分、典型问题、影响范围、整改建议和复核结果，形成可放入盘点报告的质量分析章节。

### 【输出】

1. 数据质量规则匹配清单。
2. 规则推荐与人工确认清单。
3. 规则参数配置记录。
4. 质量评价任务执行记录。
5. 规则执行明细和异常样例。
6. 六维质量评分表和总分结果。
7. 质量问题清单、整改建议和闭环状态。
8. 质量问题溯源及影响范围分析。
9. 数据质量分析报告章节。

### 【AI提示词】

你是一名资深数据质量评价专家，请根据以下材料生成数据产权登记盘点报告中的“数据质量分析”部分。

【数据资源基本信息】{数据资产名称、业务域、数据来源、数据形态、数据规模、数据责任人}

【数据字典】{表名、字段名、字段类型、字段描述、示例值、是否必填、是否主键、枚举值}

【质量评价规则】{规则编号、规则名称、评价维度、适用对象、检查要求、阈值、校验级别、整改建议}

【质量评价结果】{规则通过率、失败规则、异常记录数、异常样例、任务执行批次}

【质量问题与溯源】{问题清单、责任角色、整改建议、源表源字段、ETL环节、下游影响对象}

要求：

1. 按规范性、完整性、准确性、一致性、时效性、可访问性六个维度分别说明评价情况。
2. 说明规则推荐、人工确认、参数设定和任务执行过程。
3. 对典型质量问题进行归类，说明问题字段、问题原因、影响范围和整改建议。
4. 如已接入真实血缘数据，应说明问题来源和下游影响；如暂未接入，应说明已预留溯源接口，待真实血缘数据导入后开展溯源。
5. 输出质量评分、质量等级建议和后续整改建议。
6. 结论可采用如下表述：“经质量评价，本次拟登记数据已按规范性、完整性、准确性、一致性、时效性、可访问性等维度完成检查，评价过程可追溯，问题清单和整改建议已形成，整体质量情况可支撑后续数据产权登记材料编制。”
